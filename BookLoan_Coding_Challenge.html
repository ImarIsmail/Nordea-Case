Technical Assessment

BookFlow: Library Loan Manager

Programming Exercise for Engineering Candidates

1. Overview & Goal

Welcome to our technical assessment! In this exercise, you will design and build a small core application called BookFlow. BookFlow is a lightweight library loan management service where Members can check out and return books, as well as rate their reading experiences.

This exercise has been designed to see how you design and build software. Rather than giving you strict, step-by-step specifications, we have deliberately left the implementation details open. This allows you to showcase your engineering instincts and trade-off selections. You should be ready to discuss any part of your solution during the interview.

Completion Target

This exercise is designed to take between 2 to 4 hours. Focus on delivering readable code that covers the core business requirements, rather than implementing extensive auxiliary features. Hacks and suboptimal solutions are ok, but be prepared to discuss them and how they could be improved given more time.

2. The Tech Stack

Your submission must utilize the following stack:

  • Backend Language & Framework: Java 17+ built on Spring Boot 3.x.
  • Web Layer: Spring WebFlux.
  • Database: Use an in-memory H2 Database
  • Frontend (Optional): A clean, functional dashboard written in Angular (v15+).

3. Database Design

Your application must manage and store persistent state. The system is required to track the following information, but the exact table structures, relationship mapping, and database normalization choices are completely up to you:

  • Books: Basic descriptive data for library catalogs (titles, authors, unique identifiers like ISBNs, quantities, etc.).
  • Members: Details about library patrons.
  • Loans: Records indicating which member has borrowed which book, when it was checked out, and whether/when it was returned.
  • Book Ratings: Reviews and numerical scores submitted by members for books they have read.

Please configure the database to spin up and seed automatically with representative dummy data (e.g., via schema.sql and data.sql or programmatic initialization on startup) so your application is immediately ready to run and test.

4. API Design

Design and implement a clean, expressive, and robust API. The endpoints, HTTP verbs, payload formats, and URI naming conventions are yours to decide. Your API must cover the following core business requirements:

  1. Loan a Book: A member should be able to request a loan for a specific book.
  2. Return a Book: A member should be able to return a book, updating the status of the loan and replenishing the library's physical inventory.
  3. Rate a Book: A member should be able to submit a rating (and optional short feedback) for a book.

Feel free to structure your code as you see fit, but do include at least one unit test and one integration test.

Authentication and Security

For the base requirements, you do not need to implement complex security layers. You are permitted to pass a member identifier (e.g., memberId) as an HTTP header, query parameter, or within the request body to simulate the current active user. It is sufficient to leave a small comment in the code how you would optimize this given more time.

5. Frontend Task (Optional Angular Dashboard)

To provide a complete user experience, you are encouraged to build a lightweight, single-page Angular dashboard. This interface should allow a library member to see a consolidated view of their account.

The mini-dashboard should support:

  • Current Loans: Display a list of the books currently borrowed by the active user.
  • Ratings Dashboard: Show a view of these books along with rating statistics:
    • The active member's own rating for that book (if they have rated it).
    • The consolidated aggregate rating of the book calculated from all members in the system.

Keep the user interface clean and intuitive. You do not need complex CSS packages; a well-structured layout using vanilla CSS (Flexbox or CSS Grid) is perfect.
